# Expression Language Benchmarks

Comprehensive benchmarking system for tracking performance across all compilation phases.

## Structure

```
benchmarks/
├── data/                          # Input data and generated artifacts
│   ├── expressions.json           # Test expressions (manually maintained)
│   ├── expressions.ast.json       # Generated AST (regenerated)
│   └── expressions.bytecode.json  # Generated bytecode (regenerated)
├── results/                       # Performance tracking
│   ├── performance-history.jsonl  # Append-only performance log
│   └── performance-latest.json    # Most recent benchmark results
└── scripts/                       # Benchmark tooling
    ├── generate.ts                # Regenerate AST and bytecode
    ├── benchmark.ts               # Run benchmarks and track history
    └── compare.ts                 # Compare current vs historical performance
```

## Usage

### 1. Generate AST and Bytecode

```bash
bun run packages/core/benchmarks/scripts/generate.ts
```

Reads `data/expressions.json` and generates:
- `data/expressions.ast.json` - AST for each expression
- `data/expressions.bytecode.json` - Bytecode for each expression

### 2. Run Benchmarks

```bash
bun run packages/core/benchmarks/scripts/benchmark.ts
```

Measures performance for all expressions across 4 phases:
1. **Lexer** - Tokenization
2. **Parser** - AST generation
3. **Compiler** - Bytecode generation
4. **VM** - Execution

Results saved to:
- `results/performance-latest.json` - Current results
- `results/performance-history.jsonl` - Appended to history

Skip history update:
```bash
bun run packages/core/benchmarks/scripts/benchmark.ts --no-history
```

### 3. Compare Performance

```bash
# Compare current vs first baseline
bun run packages/core/benchmarks/scripts/compare.ts

# Compare against specific commit
bun run packages/core/benchmarks/scripts/compare.ts --baseline=abc1234
```

Shows:
- Performance changes per expression
- Performance by category
- Regressions and improvements

Exits with error code 1 if regressions detected (>5% slower).

## Workflow

### Adding New Expressions

1. Edit `data/expressions.json`:
```json
{
  "id": 20,
  "source": "data.items |> sortBy(.price)",
  "category": "pipe",
  "complexity": "medium"
}
```

2. Regenerate AST/bytecode:
```bash
bun run packages/core/benchmarks/scripts/generate.ts
```

3. Run benchmarks:
```bash
bun run packages/core/benchmarks/scripts/benchmark.ts
```

### CI Integration

```yaml
- name: Benchmark
  run: bun run packages/core/benchmarks/scripts/benchmark.ts

- name: Check for regressions
  run: bun run packages/core/benchmarks/scripts/compare.ts
```

This will fail CI if performance regresses >5%.

### Before/After Performance Testing

```bash
# Baseline (before changes)
git checkout main
bun run packages/core/benchmarks/scripts/benchmark.ts
BASELINE=$(git rev-parse HEAD | cut -c1-8)

# Make your changes
git checkout feature-branch
bun run packages/core/benchmarks/scripts/benchmark.ts

# Compare
bun run packages/core/benchmarks/scripts/compare.ts --baseline=$BASELINE
```

## Architecture Benefits

### Layered Independence

Each layer can work standalone:

**Build-time compilation** (production):
```ts
// At build time
import { tokenize, parse, compile } from 'yexp';

const bytecode = compile(parse(tokenize(expr)));
fs.writeFileSync('compiled.json', JSON.stringify(bytecode));
```

**Runtime-only** (smallest bundle):
```ts
// At runtime (only VM in bundle)
import { evaluate } from 'yexp';
import bytecode from './compiled.json';

const result = evaluate(bytecode, context);
```

### Bundle Size Comparison

| Bundle | Size | Use Case |
|--------|------|----------|
| Full (lexer + parser + compiler + VM) | ~50KB | Development, dynamic expressions |
| Runtime-only (VM) | ~15KB | Production, pre-compiled expressions |

### Performance Tracking

The JSONL history format allows:
- **Regression detection** - Alert on >5% slowdowns
- **Trend analysis** - Plot performance over time
- **Git bisect** - Find which commit introduced regression
- **Category tracking** - Monitor specific operation types

## Performance Budget

Target performance per phase (10,000 iterations avg):

| Phase | Simple | Medium | Complex |
|-------|--------|--------|---------|
| Lexer | <0.005ms | <0.010ms | <0.020ms |
| Parser | <0.005ms | <0.010ms | <0.020ms |
| Compiler | <0.005ms | <0.010ms | <0.020ms |
| VM | <0.010ms | <0.050ms | <0.200ms |
| **Total** | **<0.025ms** | **<0.080ms** | **<0.260ms** |

Regressions >5% above budget trigger CI failures.
